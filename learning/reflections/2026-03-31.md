## What Was Decided

### 1. Raise coverage bar to 97%, mandate data-testid
**Decision:** Tightened quality gate policy — coverage raised from 90% to 97% (statements/branches/functions/lines). Every interactive UI element must carry a `data-testid` attribute. PRs blocked at tech-lead review without them.

**Intended outcome:** Eliminate E2E selector brittleness, push test quality higher before deploy.
**Actual outcome:** Policy set. E2E tests still have 5 failures (network timeouts + locator issues). The policy is sound but enforcement is only as good as the test run fidelity — CI green, production E2E flaky.

**What went well:** The 97% threshold is measurable and achievable (current branch coverage: 95.72%). The data-testid mandate gives QA a concrete blocking criterion.

**What went wrong:** E2E flakiness (network timeouts, locators) means the full quality benefit isn't realized yet. The policy is in place but the test infrastructure underneath still needs work.

**Principle learned:** A policy without enforced automation is aspirational. The next step (test infra sprint ticket) is the right call — don't celebrate policy wins until the infra reliably enforces them.

### 2. Change blob access from 'public' to 'private'
**Decision:** Family media (videos, photos) should not be publicly accessible. Changed blob access to `private`.

**Intended outcome:** Media URLs require signed/authenticated access. Only family members can view family content.
**Actual outcome:** Implemented. Blob access is now private. Need to verify signed URL generation works end-to-end in production.

**What went well:** The change was surgical (one line/code path). Didn't require schema changes.

**What went wrong:** Haven't confirmed signed URL flow works in production E2E yet. Risk: broken media rendering if signed URL generation has a bug.

**Principle learned:** Security changes default to "deny" — but always verify the happy path still works after tightening restrictions. A security fix that breaks the product is a partial fix at best.

### 3. Fix dashboard to use real DB data (replace placeholder)
**Decision:** Dashboard was showing hardcoded placeholder families/stats. Replaced with real Neon Postgres queries via Drizzle ORM.

**Intended outcome:** Dashboard shows actual family data for the signed-in user.
**Actual outcome:** Commit 59c45e1 shipped to preview. Dashboard connects to real DB.

**What went well:** The fix was targeted (one component), no cascading changes needed. Placeholder data is gone from the main user-facing view.

**What went wrong:** None — this was pure bug fix work, executed cleanly.

**Principle learned:** Placeholder data in UI is a trust leak. Even in early-stage products, wire UI to real data as early as possible — it surfaces schema issues, auth bugs, and data gaps before they become production surprises.
