## Decision 3: Fixed ActivityFeed empty state CTAs to use family-scoped routes (familyId-aware)

**What was decided:** Empty state CTAs in ActivityFeed were pointing to `/app/family` (family list) instead of `/app/family/${familyId}/feed` (the correct family-scoped route). Fixed both the primary CTA and the secondary "view invites" link.

**Intended outcome:** A new family user who hits an empty feed clicks the CTA and lands on their family's feed — not a generic family picker. Navigation is contextually correct.

**Actual outcome:** Commits `96fc59d` and `1d0b96b` shipped. Both the feed empty state and events empty state now use dynamic `familyId` in routes. CTAs are warm and guidance-appropriate per the empty-states spec.

**What went well:** The fix was surgical — changed two href strings. High polish impact for minimal code. The empty states spec (`empty-states-spec.md`) was already written, so there was no debate about what the correct behavior should be.

**What went wrong:** The incorrect routes existed since the components were built. No spec enforcement caught this at implementation time — only caught now via UX review.

**Principle learned:** Route hrefs are user-facing behavior. Every navigation link should be verified against the routing spec at PR review time, not discovered by a future UX walk. Add href verification to the PR checklist.

**Intended outcome:** One source of truth for mobile navigation. All hrefs resolve to real routes. No duplicate code paths.

**Actual outcome:** `2596730` shipped. `mobile-nav.tsx` is now the single source of truth. All 6 app-shell tests pass.

**What went well:** Clean refactor — reduced duplication and fixed UX bugs in the same commit. The broken `[familyId]` hrefs meant users on mobile were literally clicking placeholder routes.

**What went wrong:** The duplicate existed for some time. This is technical debt that accumulated because no one did a "find unused/dead code" scan during implementation.
